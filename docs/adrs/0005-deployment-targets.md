# ADR 0005: 部署目标与生产化边界

## 状态

Accepted

## 背景

PRD 面向校园图书馆智能座位管理原型和初赛演示。当前仓库包含本地 PostgreSQL、Mosquitto、NestJS API、小程序骨架、设备模拟器和固件骨架，但没有生产部署配置、域名、TLS、云资源或运维策略。

## 决策

本阶段只承诺两个部署目标：

- 本地开发/联调环境：使用 pnpm、Docker Compose、本地 PostgreSQL、本地 Mosquitto、后端 API、小程序体验版或开发者工具、设备模拟器或单个真实终端。
- 初赛演示环境：围绕 1 个 ESP32-P4 智能座位终端、微信小程序体验版、后端服务、本地或受控演示网络完成演示链路。

生产化边界如下：

- 不承诺公网高可用部署。
- 不承诺正式域名、TLS 证书、云数据库、云 MQTT、日志平台、监控平台或告警平台。
- 不复用本地匿名 broker、占位 secret 或演示账号作为生产配置。
- 后续生产化必须另列任务，重新评估安全、运维、备份、扩容和合规。

## 理由

初赛目标是可演示原型，不是生产系统。明确部署边界可以避免后续任务把本地脚本、匿名 broker 或演示凭据误认为生产能力，同时保留未来生产化演进空间。

## 影响范围

- `OPS-01` 只负责本地和演示启动、seed、reset-demo 与说明。
- `QA-01` 的发布闸门以本地/初赛演示为准。
- `API-AUTH-02`、`API-AUTH-03` 和 `API-IOT-01` 需要区分 mock/演示配置与真实生产配置。
- README 和演示文档不得宣称当前仓库已具备生产部署能力。

## 回滚或替代方案

若需要提前支持云端演示，可新增部署 ADR 和 OPS 任务，明确目标平台、域名、TLS、secret 注入、数据库、MQTT 安全、备份和回滚方案。该变更不得默认扩大所有业务任务范围。
