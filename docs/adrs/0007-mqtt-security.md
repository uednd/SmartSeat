# ADR 0007: MQTT 设备认证、安全策略与匿名 Broker 边界

## 状态

Accepted

## 背景

SmartSeat 设备通信采用 MQTT。当前 infra 中 Mosquitto 允许匿名，适合本地快速联调，但 PRD 要求设备连接 MQTT 时携带认证信息，未认证设备不得接入。后续任务需要明确本地匿名 broker 与正式设备认证的边界。

## 决策

本地和初赛演示允许使用匿名 Mosquitto broker，但仅限本地开发、受控演示网络和占位联调。

正式化设备认证策略如下：

- 每个设备使用唯一 `device_id`，topic 遵循 `seat/{device_id}/...`。
- 后端维护设备注册状态和绑定座位关系。
- MQTT 配置必须支持 broker URL、client id、用户名、密码和超时阈值。
- 正式或准生产环境必须禁用匿名访问，并启用设备级凭据、ACL 或 mTLS/TLS 等安全方案。
- 设备端不得保存生产明文 secret 到仓库；本地配置只能使用占位值。

GOV-02 不实现 MQTT 连接或安全配置。`API-IOT-01`、`FW-01`、`SIM-01` 和 `OPS-01` 按本 ADR 预留配置和安全边界。

## 理由

匿名 broker 能降低本地演示成本，但不能满足正式安全要求。将匿名 broker 明确限定为本地/演示用途，可以让初赛推进，同时避免后续任务误把本地配置当作生产方案。

## 影响范围

- `API-IOT-01` 需要支持 MQTT 凭据配置和设备鉴权扩展点。
- `FW-01` 和 `SIM-01` 需要通过配置读取 broker、device_id 和凭据信息。
- `OPS-01` 说明本地匿名 broker 的边界。
- `QA-01` 记录匿名 broker 作为演示降级，不作为正式发布通过条件。

## 回滚或替代方案

若初赛环境要求更高安全性，可新增任务配置 Mosquitto 用户密码和 ACL。若设备证书方案提前确定，可新增 ADR 升级为 TLS/mTLS，但不得在本地演示配置中提交真实证书、私钥或生产密码。
