# 基于 ESP32-P4 与 uni-app 微信小程序的校园图书馆智能座位管理系统 PRD

> 文档版本：v1.1  
> 适用阶段：初赛原型设计与开发  
> 开发周期：约 2.5 个月  
> 初赛演示终端数量：1 个智能座位终端  
> 核心硬件：ESP32-P4-Function-EV-Board、7 英寸触摸显示屏、毫米波人体存在传感器、状态灯  
> 设备通信：MQTT  
> 应用侧：基于 uni-app 的微信小程序体验版  
> 管理侧：学生与管理员共用同一小程序与同一登录入口，按账号身份进入不同页面；管理功能归属于后端能力，不单独建设独立 Web 管理台

---

## 1. 项目概述

### 1.1 项目背景

校园图书馆、自习室等公共学习空间存在占座混乱、预约与实际使用状态不一致、座位资源浪费、人工巡查效率低等问题。传统管理方式难以及时发现预约未到、提前离座、超时占用、设备异常等情况，也缺少对学习行为的正向反馈。

本项目通过嵌入式智能座位终端、uni-app 微信小程序和基于 NestJS 的后端服务，构建一套可演示、可扩展的智能座位管理系统。系统围绕一个实际演示座位完成完整闭环：座位预约、到座扫码、使用计时、毫米波人体存在检测、异常识别、自动释放、管理员处理、学习数据统计和匿名排行榜。

### 1.2 产品定位

本系统定位为面向校园图书馆、自习室、实验室工位等公共座位资源的智能管理原型系统。

初赛阶段交付目标不是全馆规模化部署，而是基于 1 个智能座位终端完成端到端原型验证，证明方案在身份核验、座位状态同步、IoT 设备通信、毫米波人体存在感知、异常处理和数据统计方面具备可行性。

### 1.3 产品目标

1. 学生能够通过小程序查看座位状态、预约座位、扫码签到、续约、主动离座和查看学习统计。
2. 智能座位终端能够展示座位状态、动态二维码、当前时间、剩余时长，并控制状态灯。
3. 毫米波人体存在传感器能够辅助判断座位区域是否有人存在。
4. 后端能够通过 MQTT 接收设备心跳、传感器数据和设备事件，并向设备下发状态更新。
5. 后端能够自动识别预约未到、设备离线、疑似未预约占用、疑似提前离座、到期仍有人等异常。
6. 管理员能够在同一小程序内进入管理页面，查看座位、设备和异常事件，并手动释放座位。
7. 系统能够记录学生学习时长、到馆次数、预约未到次数，并展示匿名排行榜。

### 1.4 初赛交付范围

| 模块 | 初赛交付内容 |
|---|---|
| 智能座位终端 | 1 个 ESP32-P4 智能座位终端，完成屏幕显示、动态二维码、状态灯、毫米波人体存在检测、MQTT 心跳与数据上报 |
| uni-app 微信小程序 | 学生端页面、管理员页面、统一登录入口、角色路由、座位预约、扫码签到、续约、离座、学习统计、匿名排行榜、管理操作 |
| 后端服务（NestJS） | 用户、角色、登录模式、座位、设备、预约、签到、状态机、MQTT 接入、异常检测、学习记录、排行榜、管理接口 |
| 登录认证 | 支持微信登录模式和 OIDC 登录模式；后台配置当前启用的登录模式 |
| 设备通信 | 采用 MQTT 完成智能座位终端与后端之间的设备心跳、传感器上报、状态同步和命令下发 |
| 管理能力 | 管理员在同一小程序内查看座位状态、设备在线状态、异常事件、预约未到记录，并执行手动释放 |

## 2. 用户角色、权限与登录方式

### 2.1 用户角色

| 角色 | 说明 | 核心诉求 |
|---|---|---|
| 学生用户 | 使用小程序预约和使用座位的用户 | 查看座位、预约、扫码签到、续约、离座、查看学习数据和匿名排行榜 |
| 管理员 | 在同一小程序中进入管理页面的管理人员 | 查看座位状态、设备状态、异常事件、预约未到记录，并手动释放座位 |
| 智能座位终端 | ESP32-P4 设备 | 展示状态、刷新二维码、采集毫米波传感器数据、控制灯光、上报心跳、接收后端命令 |
| 后端系统 | 业务规则和数据中心 | 统一处理登录认证、角色识别、预约状态机、设备通信、异常检测和统计计算 |

### 2.2 统一入口与角色路由

1. 学生和管理员共用同一个 uni-app 微信小程序入口。
2. 用户进入小程序后先完成登录认证。
3. 后端根据账号身份返回用户角色。
4. 学生角色进入学生首页。
5. 管理员角色进入管理员页面。
6. 系统内尚无任何用户时，首个完成注册或登录的用户自动成为系统管理员；系统已有用户后，不允许普通用户在前端自助注册为管理员，管理员身份由后端角色配置或已有身份源映射决定。
7. 同一账号可具备多个角色；初赛演示中一个管理员测试账号进入管理员页面，一个学生测试账号进入学生页面。

### 2.3 登录模式配置

系统支持两种登录模式：微信登录模式和 OIDC 登录模式。

| 登录模式 | 使用场景 | 用户创建方式 | 身份来源 |
|---|---|---|---|
| 微信登录 | 公共图书馆、自习室、社区空间等无统一身份系统场景 | 微信一键注册/登录 | 微信 openid/unionid |
| OIDC 登录 | 学校、单位、机构已有统一账号体系场景 | 不提供注册功能，仅通过 OIDC 登录 | 本校或机构 OIDC Provider |

登录模式由管理员在后台配置。系统同一时间启用一种主登录模式。

### 2.4 微信登录规则

1. 用户点击微信登录后，小程序调用微信登录能力获取 code。
2. 后端使用 code 换取 openid。
3. 若 openid 已绑定用户，则直接登录。
4. 若 openid 未绑定用户，则自动创建用户；当系统内尚无任何用户时，首个注册/登录用户自动创建为系统管理员，否则创建为普通学生用户，实现一键注册/登录。
5. 微信登录模式适用于公共机构或无校内统一账号体系的场景。
6. 微信登录模式下仅系统首个注册/登录用户自动获得管理员权限；系统已有用户后，管理员权限必须由后台配置。

### 2.5 OIDC 登录规则

1. OIDC 模式下，小程序发起统一身份认证流程。
2. 用户通过学校或机构 OIDC Provider 完成身份验证。
3. 后端接收 OIDC 登录结果，校验 id_token 或授权码换取的用户信息。
4. 后端根据 OIDC subject、学号、工号、邮箱或用户组映射本系统用户。
5. OIDC 模式下不设计注册功能。
6. OIDC 模式下未被身份源认可的用户无法进入系统。
7. 系统内尚无任何用户时，首个成功完成 OIDC 登录并进入系统的用户自动成为系统管理员。
8. 系统已有用户后，管理员身份由 OIDC 用户组、白名单或后端角色配置决定。
9. 真实 OIDC 联调属于高风险前置项，必须在项目第一周完成最小闭环验证：小程序获取登录地址、用户完成认证、后端拿到 code 或 token、后端签发系统 token、小程序完成角色路由。
10. 小程序前端不得保存或展示 `oidc_client_secret`；OIDC Client Secret 仅允许由后端保存和使用。

### 2.6 权限控制

| 功能 | 学生 | 管理员 |
|---|---:|---:|
| 查看座位状态 | 是 | 是 |
| 预约座位 | 是 | 否 |
| 扫码签到 | 是 | 否 |

| 主动离座 | 是 | 否 |
| 查看个人学习统计 | 是 | 否 |

## 3. 系统总体架构

### 3.1 系统组成

系统由以下部分组成：

1. **智能座位终端**

   基于 ESP32-P4-Function-EV-Board，连接 7 英寸触摸显示屏、毫米波人体存在传感器和状态灯。终端负责显示座位状态、动态二维码、当前时间、剩余使用时长，读取传感器数据，控制状态灯，并通过 MQTT 与后端通信。

2. **uni-app 微信小程序体验版**

   作为学生和管理员的统一入口。学生进入学生页面，完成座位预约、扫码签到、使用管理和学习统计；管理员进入管理页面，完成座位监管、设备监管、异常处理和手动释放。

3. **后端服务（NestJS）**

   负责用户认证、角色识别、登录模式配置、座位管理、预约管理、签到校验、状态机、学习统计、匿名排行榜、异常检测、管理员操作日志和 API 服务。

4. **MQTT 服务**

   负责智能座位终端与后端之间的消息通信，承载设备心跳、传感器数据、设备事件、状态同步和控制命令。

5. **数据库**

   存储用户、角色、座位、设备、预约、二维码 token、传感器记录、异常事件、学习记录、管理员操作日志、系统配置等数据。

### 3.2 通信链路

| 通信链路 | 通信方式 | 承载内容 |
|---|---|---|
| uni-app 小程序 ↔ 后端 | HTTPS API | 登录、角色识别、座位查询、预约、扫码签到、续约、离座、统计、排行榜、管理操作 |
| 智能座位终端 ↔ MQTT 服务 ↔ 后端 | MQTT | 心跳、传感器数据、设备事件、屏幕状态、二维码 token、灯光状态、控制命令 |
| 后端 ↔ 数据库 | 数据库连接 | 业务数据持久化 |
| 后端 ↔ OIDC Provider | OIDC 标准流程 | OIDC 登录认证、用户身份校验 |
| 后端 ↔ 微信服务端 | 微信登录接口 | 微信 code 换取 openid/unionid |

### 3.3 初赛部署形态

1. 初赛部署 1 个智能座位终端。
2. 小程序使用 uni-app 开发并以微信小程序体验版运行。
3. 学生页面和管理员页面在同一小程序内实现。
4. 后端部署 API 服务、MQTT 接入服务和数据库。
5. 排行榜演示所需的多用户学习数据由真实演示记录与预置测试数据共同构成。
6. 正式上架、全馆部署、多终端批量接入不纳入初赛交付。

### 3.4 MQTT 主题规划

| 方向 | Topic | 说明 |
|---|---|---|
| 设备 → 后端 | `seat/{device_id}/heartbeat` | 设备心跳上报 |
| 设备 → 后端 | `seat/{device_id}/presence` | 毫米波人体存在状态上报 |
| 设备 → 后端 | `seat/{device_id}/event` | 设备异常、重连、传感器错误等事件 |
| 后端 → 设备 | `seat/{device_id}/display` | 屏幕显示状态、二维码 token、倒计时信息 |
| 后端 → 设备 | `seat/{device_id}/light` | 状态灯控制命令 |
| 后端 → 设备 | `seat/{device_id}/command` | 重启、刷新状态、进入维护等控制命令 |

### 3.5 MQTT 消息规则

1. 每个设备使用唯一 `device_id`。
2. 设备连接 MQTT 时携带认证信息，未认证设备不得接入。
3. 设备心跳周期为 15 秒。
4. 后端超过 75 秒未收到心跳后，将设备判定为离线。
5. 设备恢复心跳后，后端将设备标记为在线，并重新下发当前座位显示状态。
6. 心跳、传感器数据使用 QoS 0 或 QoS 1；状态切换、手动释放、维护命令使用 QoS 1。
7. 所有设备消息包含 `device_id`、`seat_id`、`timestamp`。
8. 后端是座位状态的唯一可信源，终端本地状态只作为显示与上报使用。

## 4. 全局业务规则

### 4.1 座位与设备状态定义

PRD 采用双状态模型，避免设备异常直接覆盖座位原始业务状态。设备离线、传感器异常、管理员维护主要影响可用性和展示态，不直接改写当前预约/使用业务状态。

#### 4.1.1 座位业务状态 `seat.business_status`

| 状态 | 状态码 | 含义 |
|---|---|---|
| 空闲 | FREE | 当前无有效预约 |
| 已预约 | RESERVED | 用户已预约但尚未签到 |
| 使用中 | OCCUPIED | 用户已完成扫码签到并正在使用 |
| 即将结束 | ENDING_SOON | 距离预约结束时间小于 10 分钟 |
| 待释放 | PENDING_RELEASE | 到期后仍检测到有人或等待管理员处理 |

#### 4.1.2 设备在线状态 `device.online_status`

| 状态 | 状态码 | 含义 |
|---|---|---|
| 在线 | ONLINE | 设备心跳正常 |
| 离线 | OFFLINE | 超过阈值未收到设备心跳 |

#### 4.1.3 座位可用性状态 `seat.availability_status`

| 状态 | 状态码 | 含义 | 是否可新预约 |
|---|---|---|---|
| 可用 | AVAILABLE | 座位允许发起新预约 | 是 |
| 不可用 | UNAVAILABLE | 座位暂不可发起新预约 | 否 |

#### 4.1.4 不可用原因 `seat.unavailable_reason`

| 原因 | 状态码 | 含义 |
|---|---|---|
| 设备离线 | DEVICE_OFFLINE | 设备心跳超时，前端可展示为故障/离线 |
| 传感器异常 | SENSOR_ERROR | 传感器持续异常，前端可展示为故障 |
| 管理员维护 | ADMIN_MAINTENANCE | 管理员主动设置维护，前端可展示为维护中 |

说明：

1. 小程序和终端可基于 `business_status + availability_status + unavailable_reason + online_status` 派生展示态，例如“故障/维护”。
2. 当座位处于 `OCCUPIED` 或 `RESERVED` 时，即使设备离线，也必须保留原始业务状态，只将 `availability_status` 置为 `UNAVAILABLE`，并记录 `unavailable_reason`。
3. “是否可预约”以 `availability_status` 为准，不再由单一 `seat.status` 字段承载全部语义。

### 4.2 预约状态定义

| 状态 | 状态码 | 含义 |
|---|---|---|
| 待签到 | WAITING_CHECKIN | 用户已预约，等待到座扫码 |
| 使用中 | CHECKED_IN | 用户已完成扫码签到 |
| 已完成 | FINISHED | 正常使用结束并释放 |
| 已取消 | CANCELLED | 用户在签到前主动取消或系统取消 |
| 预约未到 | NO_SHOW | 用户超过签到窗口未签到 |
| 用户释放 | USER_RELEASED | 用户主动离座释放 |
| 管理员释放 | ADMIN_RELEASED | 管理员手动释放 |
| 超时结束 | TIMEOUT_FINISHED | 到期后由系统结束 |

### 4.3 签到窗口规则

1. 用户预约成功后，系统生成签到窗口。
2. 签到窗口为预约开始前 5 分钟至预约开始后 15 分钟。
3. 超过签到窗口未扫码签到，预约状态变更为 `NO_SHOW`。
4. `NO_SHOW` 后座位自动释放为 `FREE`。
5. 后端记录该用户一次预约未到。
6. 小程序个人页展示用户本周和本月预约未到次数。
7. 管理员页面展示所有预约未到记录。

### 4.4 动态二维码规则

1. 座位终端屏幕显示动态签到二维码。
2. 二维码包含 `seat_id`、`device_id`、`token`、`timestamp`。
3. 二维码每 15 秒刷新一次。
4. token 有效期为 30 秒。
5. token 成功使用后立即失效。
6. 后端校验用户身份、座位 ID、预约记录、签到窗口、token 有效期和 token 使用状态。
7. 固定二维码不得用于签到。
8. 设备离线状态下不生成新的签到 token。

### 4.5 毫米波人体存在检测规则

1. 初赛阶段使用毫米波人体存在传感器检测座位区域是否有人存在。
2. 传感器具体型号未确定，软件按照通用“人体存在传感器”抽象接入。
3. 设备适配层输出统一字段：`PRESENT`、`ABSENT`、`UNKNOWN`、`ERROR`。
4. 传感器只用于判断人体存在，不用于识别用户身份。
5. 传感器数据不直接改变预约归属，最终状态由后端状态机和异常检测规则决定。
6. `PRESENT` 持续 60 秒后，后端判定为有人存在。
7. `ABSENT` 持续 5 分钟后，后端判定为无人存在。
8. `UNKNOWN` 或 `ERROR` 持续超过 2 分钟，后端生成传感器异常事件。
9. 设备端上报原始值字段 `raw_value`；不同型号无原始值时填空。
10. 初赛安装时通过调节安装角度和检测距离，减少相邻座位干扰。

### 4.6 占用异常检测规则

| 异常类型 | 触发条件 | 系统动作 |
|---|---|---|
| 预约未到 | 已预约但超过签到窗口未签到 | 自动释放座位，记录 `NO_SHOW`，生成事件 |
| 疑似未预约占用 | 座位为 `FREE`，毫米波持续判定有人 | 生成异常事件，管理员页面展示 |
| 疑似提前离座 | 座位为 `OCCUPIED`，毫米波持续判定无人 | 生成异常事件，小程序显示提醒，管理员页面展示 |
| 到期仍有人 | 预约到期后毫米波持续判定有人 | 座位进入 `PENDING_RELEASE`，生成异常事件 |
| 设备离线 | 超过 75 秒未收到心跳 | 设备标记为 `OFFLINE`，座位 `availability_status` 置为 `UNAVAILABLE`、`unavailable_reason` 置为 `DEVICE_OFFLINE`，禁止新预约，生成事件 |
| 传感器异常 | 传感器持续 `UNKNOWN` 或 `ERROR` | 生成传感器异常事件 |

### 4.7 管理员手动释放规则

1. 管理员在小程序管理员页面对指定座位执行手动释放。
2. 已预约未签到、使用中、待释放状态均可由管理员释放。
3. 使用中座位执行手动释放时必须填写释放原因。
4. 释放后当前预约状态变更为 `ADMIN_RELEASED`。
5. 座位业务状态变更为 `FREE`；座位可用性由管理员选择恢复为 `AVAILABLE`，或保持 `UNAVAILABLE` 并记录 `ADMIN_MAINTENANCE`。
6. 系统记录管理员操作日志，包括管理员 ID、座位 ID、预约 ID、操作时间、释放原因。
7. 后端通过 MQTT 向终端下发屏幕和灯光更新命令。

### 4.8 预约未到记录与轻量惩罚规则

1. 用户超过签到窗口未签到，记录一次预约未到。
2. 小程序个人页展示用户本周和本月预约未到次数。
3. 管理员页面展示所有用户预约未到记录。
4. 用户一周内预约未到次数达到 3 次时，小程序展示提醒：“近期预约未到次数较多，请合理预约座位。”
5. 初赛阶段不执行封禁、扣分、押金、黑名单等重惩罚动作。
6. 预约未到记录计入异常事件统计。

### 4.9 匿名排行榜规则

1. 排行榜按周统计。
2. 排行指标包括累计学习时长、到馆次数、连续学习天数。
3. 排行榜展示匿名标识，例如“匿名用户 08”。
4. 不展示真实姓名、学号、手机号、微信号、openid、OIDC subject 等敏感信息。
5. 用户可在个人页开启或关闭参与排行榜。
6. 用户关闭参与后，不在榜单中展示，但本人仍可查看个人学习统计。
7. 异常刷时长记录不计入排行榜。
8. 管理员可查看排行榜整体数据，但不通过榜单暴露学生真实身份。

## 5. 模块一：智能座位终端 PRD

### 5.1 模块定位

智能座位终端部署在座位侧，用于承载座位状态展示、动态二维码展示、灯光提示、毫米波人体存在检测和 MQTT 设备通信。初赛阶段交付 1 个完整终端。

### 5.2 硬件组成

| 硬件 | 用途 |
|---|---|
| ESP32-P4-Function-EV-Board | 主控开发板，负责屏幕显示、传感器接入、灯光控制和 MQTT 设备逻辑 |
| 7 英寸触摸显示屏 | 展示座位状态、动态二维码、当前时间、剩余时长和提醒信息 |
| 毫米波人体存在传感器 | 检测座位区域是否有人存在，具体型号未确定 |
| 状态灯/RGB LED | 显示空闲、已预约、使用中、即将结束、异常等状态 |
| 电源适配器 | 为开发板、屏幕、传感器和状态灯供电 |
| 连接线材与基础元件 | 完成传感器、灯光、电源和开发板之间的连接 |

### 5.3 终端状态显示需求

#### 5.3.1 空闲状态

屏幕显示：

- 座位编号
- 当前状态：空闲
- 当前时间
- “可预约，请在小程序选择座位”提示
- 用于识别该座位的二维码展示区域

状态灯：空闲颜色常亮。

#### 5.3.2 已预约状态

屏幕显示：

- 座位编号
- 当前状态：已预约
- 当前时间
- 动态签到二维码
- 签到截止倒计时

状态灯：预约颜色常亮或慢闪。

#### 5.3.3 使用中状态

屏幕显示：

- 座位编号
- 当前状态：使用中
- 当前时间
- 剩余预约时长
- “请在小程序续约或离座”提示

状态灯：使用中颜色常亮。

#### 5.3.4 即将结束状态

触发条件：距离预约结束时间小于 10 分钟。

屏幕显示：

- 座位编号
- 当前状态：即将结束
- 剩余时间
- “请及时续约或按时离座”提示

状态灯：提醒颜色慢闪。

#### 5.3.5 待释放状态

触发条件：预约到期后，毫米波仍持续判定有人。

屏幕显示：

- 座位编号
- 当前状态：待释放
- “使用已到期，请尽快离座或联系管理员”提示

状态灯：异常提醒颜色慢闪。

#### 5.3.6 故障/维护状态

屏幕显示：

- 座位编号
- 当前状态：维护中或设备异常
- “请更换座位或联系管理员”提示

状态灯：异常颜色常亮或熄灭。

### 5.4 动态二维码需求

| 需求项 | 执行规则 |
|---|---|
| 二维码内容 | `seat_id`、`device_id`、`token`、`timestamp` |
| 刷新周期 | 15 秒 |
| token 有效期 | 30 秒 |
| 使用限制 | 成功签到后 token 立即失效 |
| 生成来源 | 后端生成 token，并通过 MQTT 下发到设备显示 |
| 离线处理 | 设备离线或后端不可达时，不使用旧 token 完成签到 |

### 5.5 毫米波传感器接入需求

1. 终端接入毫米波人体存在传感器。
2. 传感器型号未确定，终端软件实现传感器适配层。
3. 适配层统一输出 `PRESENT`、`ABSENT`、`UNKNOWN`、`ERROR`。
4. 终端通过 MQTT 上报传感器状态。
5. 上报字段包括 `device_id`、`seat_id`、`presence_status`、`raw_value`、`sensor_status`、`timestamp`。
6. 终端本地进行基础去抖处理，后端执行最终持续时间判断。
7. 传感器读取失败时，上报 `sensor_status = ERROR`。

### 5.6 状态灯需求

| 座位状态 | 灯光表现 |
|---|---|
| 空闲 | 空闲颜色常亮 |
| 已预约 | 预约颜色常亮或慢闪 |
| 使用中 | 使用中颜色常亮 |
| 即将结束 | 提醒颜色慢闪 |
| 待释放 | 异常提醒颜色慢闪 |
| 故障/维护 | 异常颜色常亮或熄灭 |

### 5.7 MQTT 通信需求

#### 5.7.1 心跳上报

Topic：`seat/{device_id}/heartbeat`

上报周期：15 秒。

Payload 字段：

| 字段 | 说明 |
|---|---|
| device_id | 设备 ID |
| seat_id | 座位 ID |
| firmware_version | 固件版本 |
| network_status | 网络状态 |
| sensor_status | 传感器状态 |
| display_status | 当前屏幕显示状态 |
| timestamp | 设备时间戳 |

#### 5.7.2 传感器上报

Topic：`seat/{device_id}/presence`

上报周期：传感器状态变化时立即上报；无变化时每 5 秒上报一次。

Payload 字段：

| 字段 | 说明 |
|---|---|
| device_id | 设备 ID |
| seat_id | 座位 ID |
| presence_status | `PRESENT`、`ABSENT`、`UNKNOWN`、`ERROR` |
| raw_value | 原始值，型号不支持时为空 |
| timestamp | 上报时间 |

#### 5.7.3 显示状态下发

Topic：`seat/{device_id}/display`

后端下发内容：

- 座位状态
- 当前时间
- 预约剩余时间
- 签到截止时间
- 二维码 token
- 提醒文案
- 屏幕布局类型

#### 5.7.4 灯光命令下发

Topic：`seat/{device_id}/light`

后端下发内容：

- 灯光状态
- 灯光颜色标识
- 常亮或闪烁模式
- 闪烁频率

### 5.8 终端本地异常处理

| 异常 | 处理策略 |
|---|---|
| Wi-Fi 断开 | 屏幕提示网络异常，设备自动重连 |
| MQTT 断开 | 设备自动重连 MQTT，并保留当前屏幕状态 |
| 后端不可达 | 屏幕提示同步异常，不生成新的签到二维码 |
| 传感器读取失败 | 上报传感器异常，并将 `presence_status` 置为 `ERROR` |
| 时间不同步 | 从后端显示命令或网络时间同步当前时间 |

### 6.3 登录需求

1. 小程序启动后调用后端获取当前系统登录模式。
2. 登录模式为微信登录时，展示微信一键登录按钮。
3. 登录模式为 OIDC 时，展示 OIDC 登录入口。
4. 登录成功后，后端返回 token、用户 ID、角色、匿名标识、登录模式。
5. 小程序根据角色进入对应页面。
6. 普通学生不能访问管理员页面。
7. 管理员页面所有操作均携带后端登录 token，并由后端鉴权。

### 6.4 学生座位查看需求

1. 学生进入小程序后查看座位状态。
2. 初赛仅展示一个真实终端座位。
3. 页面显示座位编号、座位状态、设备状态、是否可预约。
4. 前端展示态包括空闲、已预约、使用中、即将结束、待释放、故障/维护；其中故障/维护为派生展示态，不直接等同于后端业务状态字段。
5. 仅空闲座位可进入预约流程。
6. 故障/维护座位不可预约。

### 6.5 预约需求

1. 用户选择空闲座位和预约时段。
2. 后端校验用户是否已有进行中的预约。
3. 后端校验该座位在目标时段是否存在冲突。
4. 校验通过后创建预约。
5. 小程序展示预约成功结果和签到窗口。
6. 用户同一时间只能拥有一个进行中的预约。
7. 用户可在签到前取消预约。

### 6.6 到座扫码签到需求

1. 用户到达座位后，点击“扫码签到”。
2. 小程序调用微信扫码能力扫描终端屏幕二维码。
3. 小程序向后端提交二维码 token。
4. 后端校验 token、用户、座位、预约、签到窗口。
5. 校验通过后，预约状态变更为使用中。
6. 小程序进入当前使用页。
7. 终端通过 MQTT 接收状态更新并切换为使用中。

扫码失败提示：

| 失败原因 | 提示文案 |
|---|---|
| 非本人预约 | 当前座位不属于你的预约 |
| token 过期 | 二维码已过期，请重新扫描 |
| 超过签到时间 | 已超过签到时间，预约已释放 |
| 座位故障 | 当前座位设备异常，无法签到 |
| 重复签到 | 你已完成签到，无需重复操作 |
| 未登录 | 请先登录后再扫码签到 |

### 6.7 使用中需求

当前使用页显示：

- 座位编号
- 开始时间
- 结束时间
- 剩余时间
- 续约按钮
- 主动离座按钮
- 即将结束提醒
- 异常提醒，例如疑似离座、到期提醒

### 6.8 续约需求

1. 用户在预约结束前发起续约。
2. 后端检查后续时段是否可用。
3. 后续时段无人预约时允许续约。
4. 后续时段冲突时拒绝续约。
5. 续约成功后更新预约结束时间。
6. 终端屏幕倒计时同步更新。
7. 续约失败时，小程序展示失败原因。

### 6.9 主动离座需求

1. 用户点击主动离座。
2. 小程序展示二次确认。
3. 用户确认后，后端结束当前预约。

5. 座位释放为空闲。
6. 后端通过 MQTT 下发终端屏幕和灯光状态更新。

## 7. 模块三：后端服务与管理能力 PRD

### 7.1 模块定位

后端服务采用 NestJS 作为后端框架，是系统的数据中心、规则中心、设备通信中心和权限控制中心。管理能力归属于后端系统，并通过同一 uni-app 微信小程序的管理员页面呈现，不单独建设独立 Web 管理台。

### 7.2 后端核心能力

| 能力 | 说明 |
|---|---|
| 登录模式配置 | 管理当前启用的微信登录或 OIDC 登录模式 |
| 用户与角色管理 | 管理学生用户、管理员用户、角色映射和匿名标识 |
| 座位管理 | 管理座位编号、状态、绑定设备和维护状态 |
| 预约管理 | 创建预约、取消预约、签到、续约、结束预约、释放座位 |
| 签到校验 | 校验动态二维码 token、用户预约关系和签到窗口 |
| MQTT 设备通信 | 接收心跳、传感器数据和设备事件，下发显示和灯光命令 |
| 设备管理 | 判断设备在线/离线，维护设备状态 |
| 传感器数据处理 | 接收毫米波人体存在状态并执行持续时间判断 |
| 异常检测 | 识别预约未到、未预约占用、提前离座、到期仍有人、设备离线等异常 |
| 学习统计 | 计算到馆次数、累计学习时长、连续学习天数 |
| 匿名排行榜 | 按周生成匿名榜单 |
| 管理接口 | 为小程序管理员页面提供座位、设备、异常、释放、配置等接口 |
| 操作留痕 | 记录管理员释放、维护、配置变更等关键操作 |

### 7.3 登录认证能力

#### 7.3.1 登录模式配置

后端维护系统配置项：

| 配置项 | 说明 |
|---|---|
| auth_mode | `WECHAT` 或 `OIDC` |
| oidc_issuer | OIDC Provider 地址 |
| oidc_client_id | OIDC 客户端 ID |
| oidc_client_secret | OIDC 客户端密钥，后端保存 |
| oidc_redirect_uri | OIDC 回调地址 |
| admin_mapping_rule | 管理员角色映射规则 |
| wechat_appid | 微信小程序 AppID |
| wechat_secret | 微信小程序 Secret，后端保存 |

密钥管理约束：

1. `oidc_client_secret`、`wechat_secret` 仅后端可读写，不通过小程序接口明文返回。
2. 管理员页面只允许查看密钥是否已配置，不显示真实值。

#### 7.3.2 微信登录后端流程

1. 小程序提交微信登录 code。
2. 后端调用微信接口换取 openid。
3. 后端查找 openid 对应用户。
4. 用户存在时直接签发系统 token。
5. 用户不存在时创建用户；当系统内尚无任何用户时，首个注册/登录用户创建为系统管理员，否则创建为学生用户，并签发系统 token。
6. 系统已有用户后，后端根据管理员白名单或角色配置判断是否具备管理员角色。

#### 7.3.3 OIDC 登录后端流程

1. 小程序发起 OIDC 登录请求。
2. 后端生成授权地址并返回给小程序。
3. 用户通过学校 OIDC Provider 完成认证。
4. 后端接收授权 code 或 token，并在后端完成校验或换取用户信息。
5. 后端读取 OIDC subject、学号、工号、邮箱或用户组。
6. 后端根据身份源信息查找或创建本系统映射用户。
7. 当系统内尚无任何用户时，首个成功完成 OIDC 登录并进入系统的用户自动创建为系统管理员。
8. OIDC 模式下不提供前端注册功能。
9. 系统已有用户后，后端根据用户组、白名单或角色映射规则授予学生或管理员角色。
10. 后端签发本系统登录 token 给小程序。
11. 小程序根据角色进入学生页面或管理员页面。

### 7.4 预约状态机

#### 7.4.1 正常预约状态流转

`FREE → RESERVED → OCCUPIED → FINISHED → FREE`

#### 7.4.2 预约未到状态流转

`FREE → RESERVED → NO_SHOW → FREE`

#### 7.4.3 用户主动离座状态流转

`FREE → RESERVED → OCCUPIED → USER_RELEASED → FREE`

#### 7.4.4 管理员释放状态流转

`RESERVED/OCCUPIED/PENDING_RELEASE → ADMIN_RELEASED → business_status=FREE`

如管理员同时设置维护，则额外变更 `availability_status=UNAVAILABLE`，`unavailable_reason=ADMIN_MAINTENANCE`。

#### 7.4.5 设备离线状态流转

`任意 business_status + DEVICE_OFFLINE → business_status 保持不变`

同时执行：

- `device.online_status = OFFLINE`
- `seat.availability_status = UNAVAILABLE`
- `seat.unavailable_reason = DEVICE_OFFLINE`

设备离线不直接删除、结束或覆盖已有预约/使用状态，但禁止该座位产生新预约。已有预约需要按规则由系统或管理员处理。

### 7.5 MQTT 后端处理需求

1. 后端订阅所有设备心跳 topic。
2. 后端订阅所有设备传感器 topic。
3. 后端订阅所有设备事件 topic。
4. 后端接收心跳后更新设备最近在线时间。
5. 后端接收传感器数据后写入传感器记录表，并更新座位实时存在状态。
6. 后端状态机变化后，向对应设备发布屏幕显示消息。
7. 后端状态机变化后，向对应设备发布状态灯消息。
8. 后端管理员操作后，向对应设备发布强制刷新命令。
9. 后端发现设备离线后，将座位设置为不可预约，并生成异常事件。

### 7.6 异常检测需求

后端通过定时任务和实时事件处理识别以下异常：

1. 待签到预约超过签到截止时间。
2. 空闲座位持续检测到人体存在。
3. 使用中座位持续检测不到人体存在。
4. 预约到期后仍持续检测到人体存在。
5. 设备心跳超过 75 秒未上报。
6. 传感器状态持续 `UNKNOWN` 或 `ERROR`。
7. 重复扫码、过期 token、非本人预约扫码等签到异常。

### 7.7 管理能力接口

管理员页面通过 HTTPS API 调用后端接口。后端必须执行管理员角色鉴权。

| 接口 | 功能 |

| 查询异常事件 | 获取异常事件列表和详情 |
| 处理异常事件 | 标记已处理或忽略 |
| 管理员释放座位 | 手动释放指定座位并记录原因 |
| 设置座位维护状态 | 将座位设置为维护或恢复可用 |
| 查询预约未到记录 | 查看全部 no-show 记录 |
| 查询统计数据 | 查看预约数、签到数、使用率、异常数、排行榜 |

## 9. 核心流程需求

### 9.1 登录与角色路由流程

1. 小程序启动并请求后端当前登录模式。
2. 后端返回 `WECHAT` 或 `OIDC`。
3. 小程序展示对应登录入口。
4. 用户完成登录。
5. 后端返回系统 token、用户信息和角色。
6. 小程序根据角色进入学生页面或管理员页面。
7. 管理员操作由后端再次鉴权。

### 9.2 预约流程

1. 学生进入座位页。
2. 小程序请求后端座位状态。
3. 后端返回演示座位当前状态。
4. 学生选择空闲座位和预约时段。
5. 后端校验学生是否已有进行中的预约。
6. 后端校验座位时段是否冲突。
7. 校验通过后创建预约。
8. 座位状态变更为 `RESERVED`。
9. 后端生成二维码 token。
10. 后端通过 MQTT 下发终端显示状态。
11. 终端屏幕显示已预约和动态签到二维码。

### 9.3 签到流程

1. 学生到达座位。
2. 小程序扫描终端屏幕二维码。
3. 小程序提交 token 到后端。
4. 后端校验 token 有效性。
5. 后端校验预约是否属于当前学生。
6. 后端校验是否处于签到窗口。
7. 校验通过后，预约状态变更为 `CHECKED_IN`。
8. 座位状态变更为 `OCCUPIED`。
9. 后端通过 MQTT 下发使用中屏幕状态和灯光状态。
10. 小程序进入当前使用页。

### 9.4 预约未到流程

1. 后端定时扫描 `WAITING_CHECKIN` 预约。
2. 当前时间超过签到截止时间后，预约状态变更为 `NO_SHOW`。
3. 座位释放为 `FREE`。
4. 用户预约未到次数增加。
5. 后端生成预约未到异常事件。
6. 管理员页面出现预约未到记录。
7. 后端通过 MQTT 下发空闲状态给终端。

### 9.5 使用中提前离座检测流程

1. 预约处于 `CHECKED_IN`。
2. 毫米波传感器持续上报 `ABSENT`。
3. 后端判定 `ABSENT` 持续达到 5 分钟。
4. 后端生成“疑似提前离座”异常事件。
5. 小程序当前使用页展示提醒。
6. 管理员页面显示该异常。
7. 管理员现场确认后执行手动释放或忽略异常。

### 9.6 空闲座位被占用检测流程

1. 座位处于 `FREE`。
2. 毫米波传感器持续上报 `PRESENT`。
3. 后端判定 `PRESENT` 持续达到 60 秒。
4. 后端生成“疑似未预约占用”异常事件。
5. 管理员页面显示异常。
6. 管理员处理异常并记录结果。

### 9.7 到期释放流程

1. 到达预约结束时间。
2. 后端检查是否存在有效续约。
3. 不存在续约时，后端结束预约。
4. 若毫米波判定无人，座位直接释放为 `FREE`。
5. 若毫米波判定有人，座位进入 `PENDING_RELEASE`，并生成“到期仍有人”异常事件。
6. 管理员可在管理员页面手动释放座位。
7. 后端通过 MQTT 同步终端屏幕和灯光。

### 9.8 管理员手动释放流程

1. 管理员进入管理员页面。
2. 管理员选择座位。
3. 页面展示座位状态、预约状态、人体存在状态、设备状态。
4. 管理员点击“释放座位”。
5. 管理员选择或填写释放原因。
6. 后端校验管理员权限。
7. 后端结束当前预约并释放座位。
8. 后端记录管理员操作日志。
9. 后端通过 MQTT 下发终端状态更新。
10. 小程序管理员页面刷新座位状态。

### 9.9 设备离线流程

1. 后端持续接收设备心跳。
2. 超过 75 秒未收到心跳。
3. 后端将设备标记为离线。
4. 后端将绑定座位的 `availability_status` 置为 `UNAVAILABLE`，并记录 `unavailable_reason=DEVICE_OFFLINE`，但不覆盖其 `business_status`。
5. 后端生成设备离线异常事件。
6. 小程序学生页显示该座位不可预约。
7. 小程序管理员页显示设备离线。
8. 设备恢复心跳后，后端将设备标记为在线并下发最新座位状态。

### 9.10 匿名排行榜生成流程

1. 后端每日或每次查询时统计本周学习记录。
2. 过滤无效学习记录和异常刷时长记录。
3. 按累计学习时长、到馆次数、连续学习天数生成榜单。
4. 榜单仅使用匿名标识展示。
5. 用户关闭排行榜参与后，其数据不出现在公开榜单。
6. 用户本人仍可查看个人学习统计。

## 10. 非功能性需求

### 10.1 可用性

1. 学生完成一次预约不超过 4 个主要步骤。
2. 到座扫码签到在 10 秒内完成。
3. 当前使用页清晰展示剩余时间、续约入口和离座入口。
4. 管理员页面能够在首页直接看到异常数量和设备状态。
5. 终端屏幕字体和二维码尺寸满足现场扫码演示需求。

### 10.2 可靠性

1. 后端避免重复签到、重复释放、重复学习记录。
2. 设备离线时禁止新预约。
3. 二维码 token 具备过期和一次性使用机制。
4. 传感器数据采用持续时间阈值，避免单次误报改变核心状态。
5. 管理员手动释放必须记录操作日志。
6. MQTT 断线重连后，设备重新同步后端状态。

### 10.3 安全与隐私

1. 小程序与后端通信使用 HTTPS。
2. MQTT 连接使用设备身份认证。
3. 二维码 token 不包含用户隐私数据。
4. 座位屏幕不展示学生真实身份。
5. 排行榜默认匿名。
6. openid、unionid、OIDC subject、学号、手机号等敏感信息不在前端公开展示。
7. 管理员页面必须进行角色鉴权。
8. 登录配置变更、手动释放、维护状态变更均写入管理员操作日志。

### 10.4 性能

| 指标 | 目标 |
|---|---|
| 座位列表加载 | 2 秒内返回 |
| 扫码签到校验 | 1 秒内返回结果 |
| 设备心跳周期 | 15 秒 |
| 设备离线判断 | 75 秒 |
| 终端状态更新 | 后端状态变化后 3 秒内完成显示更新 |
| 传感器 PRESENT 判定 | 持续 60 秒 |
| 传感器 ABSENT 判定 | 持续 5 分钟 |

### 10.5 可扩展性

1. 虽然初赛只演示 1 个终端，后端数据模型按多座位、多设备设计。
2. 座位、设备、传感器通过 ID 绑定，支持后续扩展。
3. MQTT topic 使用 `device_id` 进行隔离，支持后续接入多个终端。
4. 登录认证模块支持微信登录和 OIDC 登录切换。
5. 排行榜和统计逻辑按周期计算，支持日、周、月扩展。
